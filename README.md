# Сборщик ВКР PDF (parserforsibgu)

Настольное приложение на **Flutter** для поисков выпускных квалификационных работ (ВКР) в электронной библиотеке СибГУ имени ак. М. Ф. Решетнёва — [elib.pallada.sibsau.ru](https://elib.pallada.sibsau.ru/) — и **скачивания всех страниц работы одним объединённым PDF-файлом**.

Заголовок в интерфейсе: **«Сборщик ВКР PDF»**.

---

## Что делает приложение

1. **Поиск** работ по ссылке на карточку, номеру книги (`id`), названию, фамилии автора или произвольной фразе из метаданных каталога.
2. Показывает **список найденных работ** на отдельном экране.
3. По кнопке **«Собрать»** для выбранной работы **скачивает все PDF-страницы**, **склеивает** их в один файл и предлагает **сохранить** его через системный диалог «Сохранить как».

Прямой доступ к базе данных приложение не имеет: всё взаимодействие — через **HTTPS** с сайтом библиотеки.

---

## Как устроен поиск

### Два режима запроса

**1. Прямая работа по `id`**  
Если введён только числовой номер, или текст распознан как URL/фрагмент с параметром `id=…` для библиотеки (например `https://elib.pallada.sibsau.ru/book_view/?id=58323`), запрос считается **точным**: загружается одна страница карточки ВКР, из HTML извлекаются ссылки на PDF страниц списком **без общего фильтра по каталогу**.

Нормализация вставленной строки учитывает: кавычки, BOM/невидимые символы, обрезку текста после тире, выделение URL внутри длинной строки.

**2. Поиск по каталогу (текстовый запрос)**  
Если это **не** похоже на один `book_id`:

- приложение отправляет **POST** на эндпоинт каталога:  
  `https://elib.pallada.sibsau.ru/ajax/library_result`
- тело запроса — **JSON-RPC 2.0** (`method: call`, пустые `params`), заголовки приближены к браузеру (`Content-Type: application/json`, `X-Requested-With: XMLHttpRequest`, `Referer` и т.д.);
- ответ декодируется: из вложенного JSON берётся массив **`result_vkr`** — записи каталога ВКР.

Дальше **фильтрация выполняется локально в приложении** (в отдельном изоляте через `compute`, чтобы не блокировать UI):

- из каждой записи собирается одна строка из всех текстовых полей;
- запрос разбивается на **слова-токены** после нормализации;
- в результат попадают только записи, в тексте которых **все** токены встречаются как подстроки (логика «И» по словам).

**Нормализация текста для поиска:** нижний регистр; `ё` → `е`; всё, кроме букв и цифр (Unicode), заменяется пробелами; лишние пробелы сжимаются.

Из подходящей записи берутся, в частности: `book_id`, заголовок (`s200a` / `naim`), автор (`description_author`), формируется ссылка на `/book_view/` с этим `id`. У записей из каталога **список страниц сначала пустой** — он подгружается при нажатии «Собрать» (см. ниже).

### Повтор запроса

На экране результатов есть **«Повторить запрос»** — снова вызывается тот же поиск с текущей строкой запроса.

### Ошибки и повторы сети

HTTP-запросы к сайту библиотеки при неудаче **повторяются до 3 раз** с небольшой задержкой. При окончательной ошибке пользователь видит сообщение об исключении.

## Как извлекаются страницы PDF у конкретной работы

После открытия HTML карточки (`/book_view/?id=…`) парсер ищет ссылки на PDF:

- атрибуты `src` у `iframe` и `embed`;
- ссылки `a[href]` на файлы `.pdf`;
- в сыром HTML — встроенные `data:application/pdf` и пути к `.pdf` внутри скриптов (регулярные выражения).

Относительные URL **дополняются базой** `https://elib.pallada.sibsau.ru`.

Номер страницы для сортировки выводится из URL (параметры вроде `page`/`str`/«лист» и т.п., либо цифры в имени файла), иначе используется порядок следования во время разбора.

Страницы **сортируются по номеру**, затем в таком порядке участвуют в сборке.

## Как собирается итоговый PDF (`PdfBuilder`)

1. **Подготовка** — создаётся временная подпапка (см. следующий раздел).
2. **Скачивание** — для каждой `WorkPage` выполняется GET (до 3 попыток; таймаут 25 секунд). Если URI — это `data:application/pdf`, данные берутся из строки без HTTP.
3. Файлы страниц сохраняются как `page_0001.pdf`, `page_0002.pdf`, … во временную папку.
4. **Склейка** — пакет **pdf_combiner** объединяет все файлы в `merged_<id>.pdf`.
5. В UI показывается **прогресс** в процентах и текст этапов (подготовка, загрузка, склейка, готово).

После успешной сборки открывается **системный диалог сохранения** (`file_selector`): имя файла по умолчанию — **очищенный заголовок ВКР** (удалены символы, запрещённые в именах файлов Windows, пробелы сжаты, длина обрезается до 80 символов; если пусто — `vkr_document.pdf`). Пользователь выбирает **любое место на диске** — приложение копирует готовый PDF туда с помощью `File.copy`.

Если пользователь **отменил** диалог сохранения, показывается сообщение что сборка завершена, но сохранение отменено; временная папка всё равно **удаляется** после завершения сценария.

---

## Где временная папка и что с ней происходит

- Корень задаётся вызовом **`path_provider` → `getTemporaryDirectory()`** — это системный **временный каталог** приложения (на Windows обычно что-то вроде пользовательского `%TEMP%\…`, конкретный путь задаёт ОС и Flutter runtime).
- Внутри него создаётся каталог вида:

  **`elib_run_<текущее_время_в_миллисекундах>_<id_работы>`**

  там лежат скачанные `page_XXXX.pdf` и файл **`merged_<id>.pdf`**.

- После копирования в выбранный пользователем путь (или при ошибке после того как каталог уже создана) приложение пытается **рекурсивно удалить** эту подпапку. Ошибки очистки **подавляются** — на работу сохранённого файла это не влияет.

**Важно:** итоговый PDF после сохранения в выбранное место живёт уже как обычный файл пользователя; копии во временном каталоге не оставляются намеренно.

## История запросов

Компонент **`HistoryStore`** хранит до **30** последних непустых запросов в **`SharedPreferences`** (ключ к данным наподобие `request_history`): запрос сохраняется при каждом успешном старте поиска с главного экрана, дубликаты по тексту удаляются, новый элемент в начале списка.

На главном экране запросы отображаются **чипами**; можно быстро повторить или нажать **«Повторить последний запрос»**.

## Структура исходного кода

| Путь | Назначение |
|------|------------|
| `lib/main.dart` | Точка входа, тема приложения Material 3 |
| `lib/screens/home_screen.dart` | Поле запроса, история, запуск поиска, переход к результатам |
| `lib/screens/search_results_screen.dart` | Список работ, прогресс сборки PDF, сохранение файла |
| `lib/services/elib_client.dart` | HTTP, JSON каталога, парсинг HTML карточки, извлечение `book_id`, модель ошибок |
| `lib/services/pdf_builder.dart` | Загрузка страниц, склейка, прогресс |
| `lib/services/history_store.dart` | История через `SharedPreferences` |
| `lib/models/work_item.dart` | `WorkItem`, `WorkPage`, сортировка страниц |

Зависимости смысловые: `http`, `html`, `pdf_combiner`, `file_selector`, `path_provider`, `shared_preferences`, `path`.

## Запуск и сборка

Требования: установленный **Flutter SDK** (см. [документацию Flutter](https://docs.flutter.dev)).

```bash
cd parserforsibgu
flutter pub get
flutter run
```

Сборка под конкретную платформу — стандартно (`flutter build windows`, `flutter build apk` и т.д.).

---

## Ограничения и надёжность

- Работа зависит от **неизменности** верстки и API сайта электронной библиотеки; при сильных изменениях на сервере парсинг HTML или разбор каталога может потребовать правок в **`ElibClient`**.
- Для сборки нужен доступ в интернет ко всем URL PDF (включая возможные ограничения сетью/ файрволом).
- Изменение структуры PDF на сайте может повлиять на успешность **pdf_combiner** для отдельных работ.
